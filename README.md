# CHOLOGBOOK | 초록북 🌱

작은 행동(Patch)을 쌓고,

그 흐름(Minor)을 인지하며,

변화(Major)를 발견하는 과정.

초록북은 **Patch → Minor → Major** 흐름을 데이터와 UX로 풀어 가는 기록 시스템입니다.  
기록을 대신 채워 주는 앱이 아니라, **스스로 변화를 느낄 수 있는 타이밍**을 만들어 주는 쪽을 지향합니다.

## 설계 목표

- **최소한의 입력**으로 기록할 수 있을 것
- **반복 가능한** 구조
- 데이터가 **자연스럽게 누적**되는 구조
- 그 누적이 **변화 인식**으로 이어질 수 있는 구조

행동이 잠시 이어지게 만드는 것을 넘어, **지속되는 흐름**을 목표로 합니다.

## 전체 구조

```
UI (사용자 행동) → State / Logic (흐름 관리) → Storage (데이터 저장)
```

## Patch · Minor · Major (설계 개념)

시스템은 세 단계를 **자동으로 채우지 않습니다.**  
임계값에 도달하면 **안내와 선택지**를 제공하고, **사용자가 인지하고 선택했을 때** 다음 레이어가 쌓이도록 설계합니다.

| 단계 | 역할 | 특징 |
|------|------|------|
| **Patch** | 가장 작은 실행 기록 | 버튼 등으로 남기는 일일 단위 기록. 누적의 기본 |
| **Minor** | 실행에 대한 피드백 기록 | 느낌·개선·변화 인식. **사용자 선택**으로 생성 |
| **Major** | 변화에 대한 정리된 글 | Patch·Minor의 맥락을 **사용자가 한 편의 글로** 정리 |

핵심 흐름: **Patch (행동) → Minor (피드백) → Major (정리)**  
강요하지 않고, **선택은 항상 사용자**에게 둡니다.

> 구현은 이 설계를 기준으로 **단계적으로** 맞춰 가고 있습니다. (예: Patch 중심 동작부터 확장)

## 도메인 모델

### Topic

반복하고 싶은 행동 단위. 하나의 루틴(흐름)을 정의합니다.

```ts
type Topic = {
  id: string;
  title: string;
};
```

### Log (Patch 단위)

가장 작은 실행 기록. 저장·조회의 기본 단위는 **Log**입니다.

```ts
type Log = {
  id: string;
  userId: string;
  topicId: string;
  date: string; // YYYY-MM-DD
  text: string;
};
```

---

## 데이터 구조

### 단일 `logs`

모든 기록은 하나의 목록으로 관리합니다.

```ts
logs: Log[];
```

### Topic별 접근

중첩 구조 대신, 필요할 때 필터합니다.

```ts
getLogsByTopic(logs, topicId);
```

이렇게 두면 구조가 단순해지고, 데이터 흐름과 확장(분석 등)이 다루기 쉬워집니다.

### 기록 추가 (`addLog`)

1. 중복 검사 (`topicId` + `date`)
2. 새 `Log` 생성
3. 상태 반영
4. Firestore 저장 (비동기)

특징: **하루 1회** 기록 제한, 중복 방지, UI는 가능한 한 즉시 반영.

---

## 상태 관리 원칙

- **`logs`**: 전체 데이터의 단일 기준
- **`topics`**: 화면·그룹 기준 데이터
- **파생 값은 상태가 아니라 계산**  
  예: 토픽별 로그 묶음, 연속 일수(Streak) 등은 `logs`에서 derive

---

## Storage (Firebase)

### Firestore

- **`logs`**: 위 `Log` 필드 + `createdAt` (서버 타임스탬프)
- **`topics`**: Topic 메타(제목 등) 동기화 시 사용

데이터는 **`userId` 기준**으로 분리됩니다.

---

## 인증

- **익명 로그인**: 진입 장벽을 낮춤
- **Google 로그인**: 같은 기기라도 계정 전환 시 **새 uid 기준**으로 데이터가 갈라질 수 있음 (UI에서 안내)

---

## UX 방향

- **최소 행동**: 예) “오늘도 했어요” 한 번으로 오늘의 Patch
- **상태 기반 UI**: 익명으로 바로 사용 / 로그인 시 데이터 유지(계정 단위)
- **타이밍**: Patch가 쌓인 뒤 Minor·Major를 **제안**하되, 생성은 사용자 선택에 맡김

## 트레이드오프 (설계 선택)

| 선택 | 이유 | 결과 |
|------|------|------|
| 단일 `logs` 구조 | 단순화 | 토픽별 필터 비용 |
| 익명 로그인 | 빠른 시작 | 계정 전환 시 데이터 단절 가능 |
| 하루 1회 제한 | 기록의 의미 유지 | 같은 날 여러 번 남기기 어려움 |
